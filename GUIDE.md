# The full guide - what every piece is and why it exists

Companion to [CHEATSHEET.md](CHEATSHEET.md) (that one = the commands, this one = the understanding).
Based on kunchenguid's setup from https://youtu.be/5N-okeDdIuI, translated to Windows 11 + WSL.

## The big picture

"Dotfiles" = your machine's configuration, kept as text files in a git repo
instead of scattered settings you'd have to redo by hand on a new computer.
The payoff: a new machine becomes `git clone` + one command. Your setup is
split in two because you live in two worlds:

- **[dotfiles](https://github.com/RamiroRivas2/dotfiles)** (this repo) - the
  Windows half: WezTerm, fonts, PowerShell, winget-installed programs.
- **[dotfiles-wsl](https://github.com/RamiroRivas2/dotfiles-wsl)** - the Linux
  half: your Ubuntu inside WSL, managed with real Nix + home-manager exactly
  like the video.

Rule of thumb: things with a *window* (terminal app, fonts, dictation) belong
to Windows; things that are *pure command line* (zsh, tmux, agents at work)
live in WSL.

## Layer 1 - the foundation

### WezTerm (the window everything lives in)
A terminal emulator: the program that draws the text window your shells run
inside. Chosen because it's cross-platform (same app as the video), GPU-fast,
and configured in a real programming language (Lua) so the config lives in
the repo. Ours: rose-pine-moon colors, Hack Nerd Font, 80% opacity with
Windows acrylic blur, tab bar with integrated close buttons (our Windows fix -
the video hides all window chrome), starts PowerShell 7, one commented line
switches it to open WSL directly. Config: `home\wezterm\wezterm.lua` - edits
apply instantly.

### Shells: PowerShell 7 (Windows) and zsh (WSL)
The shell is the program that reads what you type. The video uses zsh; on
Windows your equivalent is PowerShell 7. Both are configured to feel
identical: same aliases, same ghost-text suggestions from history (Ctrl+F
accepts), same green-when-valid command coloring, same prompt. PowerShell
gets this from `home\powershell\profile.ps1`; zsh is generated by
home-manager from `home.nix`.

### starship (the prompt)
The prompt is the text printed before your cursor. starship is a fast,
shell-independent prompt engine - one config gives the identical prompt in
PowerShell AND zsh. Ours shows: current folder, git branch + dirty marker,
duration of slow commands, then `❯` (red if the last command failed). We
changed Kun's two-line layout to a single line. Config: `home\starship.toml`
(Windows) / the starship block in `home.nix` (WSL).

### Hack Nerd Font
A monospace font "patched" with thousands of extra icons (git symbols,
folder icons, spinners). Terminal tools assume these glyphs exist - without
a Nerd Font you see empty boxes. Installed per-user by `bootstrap.ps1`
straight from the nerd-fonts GitHub releases (it's not on winget).

### Package managers: winget vs Nix + home-manager
This is the deepest difference between your machine and the video.
- **Windows:** `winget` (built into Windows) installs programs;
  `bootstrap.ps1` is the list of what to install plus the wiring of configs.
  Additive: removing a line stops installing, doesn't uninstall.
- **WSL:** **Nix** is a package manager where the entire machine state is
  *declared* in text (`home.nix`). **home-manager** is the layer that applies
  it to your user: packages, zsh config, prompt, symlinks - one
  `./rebuild.sh` makes reality match the file, including *removing* things
  you delete. This is the "reproducibility" the first chapter of the video
  is about. **nix-darwin** (his Mac system layer) has no Windows equivalent
  and you don't need it - `windows-settings.ps1` covers the same ground.

### The CLI toolbelt (installed both sides)
- **ripgrep (`rg`)** - search text across thousands of files instantly.
  Powers Neovim's Space+s.
- **fd** - fast file finder. Powers Neovim's Space+f.
- **fzf** - interactive fuzzy picker for anything.
- **jq** - reads/transforms JSON on the command line (your Claude statusline
  on Mac would use it; ours uses PowerShell).
- **lazygit** - full git UI in the terminal: stage, commit, push, branch,
  resolve - all visual. Type `lazygit` in any repo, `q` quits.

### herdr (WSL) - the agent multiplexer (kun's actual tool, video ch. 33:22)
A terminal multiplexer like tmux - one window holds many shells, sessions
survive closing the window - but *agent-aware*: a sidebar shows each running
agent's state (working / blocked / done), so you can run several Claudes and
see who needs attention. Prefix set to **Ctrl+a**; **Ctrl+a Shift+A** opens
Claude Code in a new pane. Start/reattach with `herdr`. Config:
`home/herdr.toml` in the WSL repo.

### tmux (WSL) - kept as firstmate's engine
The classic multiplexer herdr replaces for daily use. It stays installed
because firstmate drives tmux as its verified backend (herdr support there is
experimental). Config: `home/tmux.conf` (prefix Ctrl+a, Ctrl+a A = Claude +
nvim layout) if you ever drive it directly.

### Neovim (the editor)
A modern vim: edits text with composable keyboard commands, extended by Lua
plugins. Identical config both sides (separate copies - sync changes!).
- **lazy.nvim** - the plugin manager; reads the plugin list in
  `home/nvim/lua/plugins/` and installs from GitHub automatically.
- **snacks.nvim** - the pickers: Space+f (files), Space+s (text), Space+b
  (buffers), plus nicer notifications.
- **oil.nvim** - file browser as an editable text list (Space+e): rename by
  editing a line, delete with dd, `:w` applies for real.
- **neogit + diffview** - git inside the editor (Space+g): s stage,
  cc commit, P p push.
- **gitsigns** - the gray "who wrote this line, when" text and change
  markers in the gutter, automatic.
- **which-key** - press Space, wait: menu of everything. Your training wheels.
- **plenary** - a library the others depend on; not user-facing.
Settings: 2-space indent, relative line numbers, system clipboard shared,
Esc also saves.

### git + GitHub CLI (`gh`)
git tracks every change to your repos; `gh` talks to GitHub from the
terminal (create repos, PRs, auth). Installed and logged in on both sides.
Also the auth backbone for gh-axi and no-mistakes' PR step.

### Claude Code + the shared AGENTS.md
The AI agent CLI, installed on Windows and (via Nix!) in WSL. One
`home/AGENTS.md` file holds your standing instructions and is linked/copied
to Claude (`~/.claude/CLAUDE.md`), Codex, and opencode - one policy, every
agent, version-controlled. The `cc` alias runs Claude with permission
prompts OFF (fast but trusting - be deliberate). A custom statusline
(`home/claude/statusline.ps1`) shows model + context usage. Windows note:
after editing AGENTS.md run `rebuild.ps1` (files are copies because
symlinks need Developer Mode).

## Layer 2 - the agent workflow tools (the video description list)

### skills CLI (vercel-labs) - app store for agent abilities
"Skills" are folders of instructions that teach Claude new tricks. This npx
tool installs them from GitHub into the right place: `npx skills find`
(browse), `npx skills add owner/repo -g` (install), `npx skills list`.
Nothing to install - runs on demand via Node.

### lavish - visual answers you can mark up
Claude writes an HTML page instead of a wall of text; it opens in your
browser; you click any element and attach a note; Claude reads your notes
and revises. Installed as a skill both sides. Trigger phrase: "show me X
**as a lavish artifact**". Your cheat sheet was delivered this way.

### no-mistakes - the quality gate
A git proxy: `no-mistakes init` once per repo, then `git push no-mistakes`
runs an AI pipeline (review, test, fix, docs, lint) in a throwaway worktree
BEFORE anything reaches GitHub, then opens a clean PR. Kun's answer to "AI
slop in my repos". Runs a small background daemon (installed both sides).
Health check: `no-mistakes doctor`.

### gnhf ("good night, have fun") - overnight agent loops
You give an objective (`gnhf "make all tests pass"`); it runs Claude in a
loop, committing every improvement, rolling back failures, keeping notes
between rounds, showing a status TUI. First times: run on a branch
(`git switch -c experiment`) until you trust it - it commits autonomously.

### AXI / gh-axi - a GitHub CLI built for agents
AXI is Kun's design idea: CLIs made for agents (compact output, clear
errors) beat both raw CLIs and MCP servers. gh-axi is the reference tool -
when Claude does GitHub work, say "use gh-axi" and it burns fewer tokens
with fewer mistakes. Installed both sides; it's for Claude, not for you.

### treehouse - parallel workspace pool (WSL)
Keeps a pool of ready-to-go copies of your repo (git worktrees) so several
agents can work simultaneously without colliding or re-downloading
dependencies. You never run it directly - firstmate uses it.

### firstmate - the crew (WSL, the deep end)
Orchestration: you talk to ONE agent, it spawns worker agents in visible
tmux panes, each in its own treehouse worktree, and supervises them to
finished PRs. Cloned at `~/github/firstmate`; start inside tmux with
`cd ~/github/firstmate && claude`. Try it last - it changes how you work.

### OpenSuperWhisper - the one that couldn't come
Voice dictation for prompts, macOS-Apple-Silicon-only (Swift app). Your
substitutes: **Win+H** (built into Windows, zero install) or **Handy**
(open-source, offline Whisper, cross-platform) if dictation sticks.

## The glue (how configs actually reach their programs)

| Mechanism | Used for | Why |
|---|---|---|
| `$PROFILE` dot-sources repo file | PowerShell profile | no admin needed, edit-in-place |
| `WEZTERM_CONFIG_FILE` env var + `~/.wezterm.lua` stub | WezTerm | env var is primary, stub catches launches that miss it |
| `STARSHIP_CONFIG` env var (+ set in profile) | starship (Win) | starship reads it every prompt |
| Directory junction | `%LOCALAPPDATA%\nvim` → repo | directory symlink that needs no admin |
| Copies + `rebuild.ps1` | AGENTS.md targets (Win) | real symlinks need Developer Mode |
| home-manager symlinks | everything in WSL | the Nix way; `mkOutOfStoreSymlink` keeps them editable |
| Backups | anything replaced | `*.backup-<timestamp>`, never deleted |

Everything ends in the same loop: **edit file in repo → apply (often
automatic) → commit → push.** That loop IS the setup.
