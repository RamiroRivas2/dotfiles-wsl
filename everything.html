0 · Open a terminal & know where you are

  1. Win key → type wezterm → Enter. The purple window = Windows / PowerShell, folders like C:\Users\ramir\...
  2. Type wsl Enter → now you're in Ubuntu / Linux, folders like /home/ramiro/..., projects in ~/github. exit goes back.
  3. Lost? pwd Enter: answer starts C:\ = Windows, starts /home = Ubuntu. The agent tools (herdr, no-mistakes, gnhf, treehouse, firstmate) live in Ubuntu.

Two dotfiles repos rule both worlds: Documents\GitHub\dotfiles (Windows) and ~/github/dotfiles-wsl (Ubuntu, real Nix). Edit repo → apply → add → git commit → push. That loop IS the setup.

2 · herdr WSL

Effort, ultracode, and thinking - the power dials

Effort levelHow hard I think per answer: low → medium → high → xhigh → max. Yours is already max - set in C:\Users\ramir\.claude\settings.json as "effortLevel": "max". Change it there (or per-tool: some skills accept a level, e.g. /code-review high).
UltracodeThe multi-agent mode - instead of working alone I spawn a fleet of worker agents that research/build/verify in parallel (this doc's command tables came from 8 agents interrogating your real tools). Your sessions have it ON. Per-message: include the word ultracode in a prompt to request it; it costs a lot more usage, so it's for big jobs (audits, migrations, deep research), not quick questions.
Token budget hintsAdd something like +300k to a message to give a big job an explicit usage target to work within.
Extended thinkingSay think hard (or ultrathink) in a prompt for extra-deep reasoning on a gnarly problem before I act.
Plan modeShift+Tab twice - I explore the code and present a plan; nothing gets edited until you approve. Use for scary changes.

Global rulesyour home\AGENTS.md → ~\.claude\CLAUDE.md - standing orders for EVERY session (no em dashes, no co-author lines...). Edit it, run rebuild.ps1, commit.
Per-project rulesCLAUDE.md in a repo (make one with /init) - project-specific conventions

Patterns that make me most useful

  • Screenshot bugs instead of describing them - paste the image, say "this looks wrong"
  • One tab/workspace per project - context stays clean; claude -c picks up where each left off
  • Delegate the goal, not the steps - "make the invoice page export CSV" beats a list of file edits
  • Say what happened, paste the exact error - I debug from real output, not summaries
  • Teach me once - repeated preference? #note it or put it in AGENTS.md, never say it twice
  • Let me verify - ask me to run the app/tests after changes; "done" should mean proven, not hoped

6 · Flows - from double-click to done

Flow 1 · Daily driver

  1. WezTerm → wsl → cd ~/github/<project> → herdr
  2. Ctrl+a Shift+A = Claude · Ctrl+a v then nvim . = the Kun layout
  3. Leave: Ctrl+a q (all keeps running) · Return: herdr

First time only: herdr integration install claude, and log in to Claude when it asks.

Flow 1b · The cockpit - where do I type what?

Every pane is just a terminal. Claude gets one, nvim gets one, and YOUR commands (git, no-mistakes...) go in a third plain pane - never into Claude's chat box or into nvim. Being "in the repo" just means the pane's folder is the repo (cd ~/github/<repo> before starting herdr; new panes inherit it).

┌─────────────┬─────────────┐
│             │    nvim     │  ← read code   (Ctrl+a v, then: nvim .)
│   Claude    ├─────────────┤
│  (talk)     │   shell ❯   │  ← run commands (Ctrl+a -, stays a shell)
└─────────────┴─────────────┘

Run no-mistakes attach in the shell pane and zoom it with Ctrl+a z. Or skip typing entirely: tell the Claude pane "commit this and push it through no-mistakes" - it runs the commands itself. Bonus: in Claude's chat, a line starting with ! runs directly in the shell (e.g. !no-mistakes daemon status).

Flow 2 · Ship a change through no-mistakes

  1. First time: no-mistakes daemon start → cd ~/github/dotfiles-wsl → no-mistakes init
  2. git switch -c my-change → edit (nvim or Claude) → add → git commit -m "..."
  3. git push no-mistakes my-change → no-mistakes attach to watch review→test→fix run
  4. Pipeline opens the PR on GitHub → review → Merge → locally: m then pull

✅ when a PR appears you never created by hand. Nothing happening? no-mistakes daemon status.

Flow 3 · Ask Claude for work, verify, ship

  1. In the repo: git switch -c fix-thing → claude (or cc)
  2. Describe the GOAL in plain English → let it work → YOU verify (run it, or browse the diff in lazygit)
  3. Happy: add → git commit -m "..." → push (or the no-mistakes gate). Unhappy: tell it what's wrong, or discard: git checkout .

Flow 4 · Overnight gnhf

  1. Prep once: Windows sleep = Never (plugged in) · check Claude credits · daytime trial with --max-iterations 2
  2. In herdr, clean tree (git status), then: gnhf --max-iterations 25 --stop-when "tests pass" "objective"
  3. Ctrl+a q, sleep. Morning: herdr → cat .gnhf/runs/*/notes.md → git log --oneline main..gnhf/... → merge / cherry-pick / delete

Flow 5 · Change your setup (example: new alias)

Windows: nvim home\powershell\profile.ps1 → add function st { git status } → new terminal. WSL: nvim home.nix → add st = "git status"; to shellAliases → ./rebuild.sh. Both: add → commit → push.

Flow 6 · First firstmate voyage

tmux new -s firstmate → cd ~/github/firstmate && claude → approve tasks-axi → "ahoy! scout my github project RamiroRivas2/InventorySystem - report on how invoices are generated" → watch via Ctrl+a w (don't type in crew windows) → read the report → for ship tasks, reply "merge it" when happy.

Every flow ends the same: add → git commit -m "..." → push. Do that and no mistake is permanent.
